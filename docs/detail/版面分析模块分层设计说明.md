# 版面分析模块分层设计说明

## 1. 设计目标

版面分析模块的目标不是直接生成 EPUB，而是把页面图片转换成稳定、可解释、可追溯的结构化版面数据，为后续 OCR、阅读顺序恢复、文本清洗、图文摘要和 EPUB 章节组织提供统一输入。

针对目标文档类型，本模块需要同时覆盖：

- 杂志类 PDF，例如《中国国家地理》：多栏、图文混排、跨页图、图注、侧栏、装饰标题较多。
- 中英文教材：标题层级、正文、例题、图表、公式、页眉页脚、边注较多。
- 学术论文：双栏正文、摘要、图表、公式、脚注、参考文献较多。

因此不能只依赖单一模型，也不能只靠规则。推荐采用三层架构：

```text
第一层：PP-DocLayout 版面检测
  -> 给出页面中的基础区域：标题、正文、图片、表格、公式、页眉页脚等

第二层：自定义规则引擎
  -> 修正模型输出，判断阅读顺序、栏位、页眉页脚、标题层级、复杂度

第三层：视觉大模型兜底
  -> 只处理复杂页面，辅助判断阅读顺序和生成图片/图表说明
```

## 2. 分层设计具体含义

“分层”不是把同一件事重复做三遍，而是让每一层解决自己最擅长的问题。

### 2.1 第一层：模型负责“看见页面上有什么”

PP-DocLayout 负责检测页面上的版面区域。

它的核心职责是：

- 找出页面上有哪些区域。
- 给出每个区域的坐标。
- 给出区域的初步类型。
- 给出检测置信度。

例如：

```text
这块像标题
这块像正文
这块像图片
这块像表格
这块像页眉
```

第一层不负责最终阅读顺序，也不负责决定 EPUB 中哪些内容先后出现。模型输出只是“候选版面结果”。

### 2.2 第二层：规则负责“整理成可读结构”

自定义规则引擎负责把模型结果整理成适合电子书转换的结构。

它的核心职责是：

- 判断页面是单栏、双栏、多栏还是复杂图文页。
- 给每个区域分配阅读顺序。
- 修正模型误判，例如把页眉误判为标题，把图注误判为正文。
- 标记页眉、页脚、页码，方便后续清洗。
- 判断页面复杂度，决定是否需要调用视觉大模型。

规则层是整个系统能否稳定输出 EPUB 的关键。模型只给“局部判断”，规则层负责“文档阅读逻辑”。

### 2.3 第三层：视觉大模型负责“复杂页面兜底”

视觉大模型不应该每页都调用，只处理传统模型和规则不确定的页面。

它的核心职责是：

- 判断复杂杂志页的阅读顺序。
- 判断图文混排页面中哪些文字属于正文，哪些属于图注、侧栏、装饰文字。
- 为图片、地图、图表生成简短说明。
- 对规则层低置信度页面给出结构建议。

第三层不是主流程，而是低频兜底模块。这样可以控制成本和处理时间，也能避免所有页面都依赖大模型的不稳定输出。

## 3. 总体数据流

```text
pages/page_001.png
  |
  v
第一层：PP-DocLayout 检测
  |
  v
raw_layout/page_001.json
  |
  v
第二层：规则引擎修正
  |
  v
layout/page_001.json
  |
  +-- 页面简单/中等复杂 -> 直接进入 OCR 和清洗
  |
  +-- 页面复杂/低置信度 -> 调用视觉大模型
                                |
                                v
                              vlm/page_001.json
                                |
                                v
                              合并修正 layout/page_001.json
```

最终下游模块只读取统一后的 `layout/page_001.json`，不直接依赖 PP-DocLayout 原始输出或大模型原始输出。

## 4. 推荐目录结构

```text
storage/
  tasks/
    {task_id}/
      pages/
        page_001.png
        page_002.png
      layout_raw/
        page_001.json
        page_002.json
      layout/
        page_001.json
        page_002.json
      vlm/
        page_001.request.json
        page_001.response.json
      debug/
        page_001_layout_overlay.png
```

说明：

- `layout_raw/` 保存第一层模型原始输出，便于排查问题。
- `layout/` 保存规则修正后的统一结果，是下游唯一依赖的数据。
- `vlm/` 保存复杂页调用大模型的请求和响应，便于复现。
- `debug/` 保存带检测框的调试图片，便于人工检查。

## 5. 统一输出 JSON 设计

### 5.1 页面级结构

```json
{
  "task_id": "demo_task",
  "page_no": 1,
  "image_path": "pages/page_001.png",
  "width": 2480,
  "height": 3508,
  "page_type": "magazine_complex",
  "layout_type": "multi_column",
  "complexity": {
    "level": "high",
    "score": 0.82,
    "reasons": [
      "figure_area_ratio_gt_0.4",
      "text_blocks_scattered",
      "more_than_two_columns"
    ],
    "need_vlm": true
  },
  "blocks": []
}
```

### 5.2 区块结构

```json
{
  "block_id": "p001_b0001",
  "source": "rule_corrected",
  "raw_type": "text",
  "role": "body",
  "text": "",
  "bbox": {
    "x1": 184,
    "y1": 420,
    "x2": 1120,
    "y2": 860
  },
  "confidence": {
    "detector": 0.91,
    "rule": 0.86,
    "final": 0.88
  },
  "column": 1,
  "order": 7,
  "reading_group": "main",
  "is_noise": false,
  "notes": []
}
```

### 5.3 role 枚举

建议第一版统一成以下类型：

| role | 含义 | 后续处理 |
|---|---|---|
| `title` | 主标题或章节标题 | 参与目录和章节识别 |
| `subtitle` | 副标题 | 可进入章节开头 |
| `body` | 正文 | 进入 OCR 和 EPUB 正文 |
| `caption` | 图注、表注 | 可保留在图片说明附近 |
| `figure` | 图片、地图、照片 | P2 阶段生成摘要或保留占位 |
| `table` | 表格 | P2 阶段做表格识别或摘要 |
| `formula` | 公式 | 可保留占位或截图 |
| `header` | 页眉 | 后续默认过滤 |
| `footer` | 页脚 | 后续默认过滤 |
| `page_number` | 页码 | 后续默认过滤 |
| `sidebar` | 边栏、补充阅读 | 可选进入正文或附注 |
| `adornment` | 装饰性文字 | 默认不进入正文 |
| `unknown` | 不确定区域 | 低置信度保留，必要时人工检查 |

## 6. 第一层：PP-DocLayout 检测

### 6.1 输入

每页一张图片：

```text
storage/tasks/{task_id}/pages/page_001.png
```

要求：

- 图片方向正确。
- DPI 建议 200-300。
- 页面尺寸已记录。
- 文件名按页码排序。

### 6.2 输出

第一层输出原始检测结果，不做复杂业务判断：

```json
{
  "page_no": 1,
  "image_path": "pages/page_001.png",
  "raw_blocks": [
    {
      "raw_id": "raw_001",
      "raw_type": "title",
      "bbox": [120, 86, 1840, 180],
      "score": 0.93
    }
  ]
}
```

### 6.3 类型映射

PP-DocLayout 的原始类型需要映射到系统内部 `role`：

| 模型类型 | 内部 role |
|---|---|
| `title` | `title` |
| `text` | `body` |
| `plain text` | `body` |
| `figure` | `figure` |
| `table` | `table` |
| `caption` | `caption` |
| `formula` | `formula` |
| `header` | `header` |
| `footer` | `footer` |
| 无法识别 | `unknown` |

注意：映射只是初步结果，第二层规则可以继续修正。

### 6.4 第一层不做的事

第一层不要承担以下职责：

- 不决定最终阅读顺序。
- 不直接删除页眉页脚。
- 不直接把文本写入 EPUB。
- 不直接判断章节层级。
- 不直接调用大模型。

## 7. 第二层：自定义规则引擎

第二层是核心工程层。它接收第一层结果，输出下游可用的统一版面 JSON。

### 7.1 规则层处理流程

```text
读取 raw_layout
  |
  v
坐标标准化
  |
  v
类型初步映射
  |
  v
页面区域统计
  |
  v
单栏/双栏/多栏/复杂页判断
  |
  v
页眉页脚页码修正
  |
  v
标题/正文/图注修正
  |
  v
阅读顺序排序
  |
  v
复杂度评分
  |
  v
输出 layout/page_001.json
```

### 7.2 坐标标准化

所有区块统一使用左上角坐标系：

```json
{
  "x1": 100,
  "y1": 200,
  "x2": 800,
  "y2": 360
}
```

同时计算辅助字段：

```text
width = x2 - x1
height = y2 - y1
area = width * height
center_x = (x1 + x2) / 2
center_y = (y1 + y2) / 2
area_ratio = area / page_area
```

这些字段不一定写入最终 JSON，但规则层内部应该使用。

### 7.3 页面类型判断

建议先定义 5 类页面：

| page_type | 说明 |
|---|---|
| `book_text` | 普通书籍/教材正文页 |
| `paper` | 论文页，通常双栏、图表较规整 |
| `magazine_simple` | 简单杂志页，图文混排但顺序清楚 |
| `magazine_complex` | 复杂杂志页，多栏、大图、侧栏、装饰文字明显 |
| `unknown` | 不确定 |

第一版可以不追求分类百分百准确，主要用它辅助后续排序策略和复杂页判定。

### 7.4 栏位判断

规则思路：

- 只用正文候选块参与判断，不用图片、页眉、页脚、页码。
- 统计文本块的 `center_x` 分布。
- 如果正文块集中在一个主要 x 区间，判为单栏。
- 如果集中在两个主要 x 区间，判为双栏。
- 如果分布在三个或更多区间，判为多栏或复杂页。

输出：

```json
{
  "layout_type": "double_column"
}
```

区块级输出：

```json
{
  "column": 1
}
```

建议枚举：

| layout_type | 含义 |
|---|---|
| `single_column` | 单栏 |
| `double_column` | 双栏 |
| `multi_column` | 多栏 |
| `image_dominant` | 图片主导 |
| `mixed_complex` | 复杂混排 |
| `unknown` | 不确定 |

### 7.5 页眉、页脚、页码修正

不能只看模型类型，要结合坐标和重复性。

候选规则：

- 位于页面顶部 5%-8% 且高度较小的短文本，候选 `header`。
- 位于页面底部 5%-8% 且高度较小的短文本，候选 `footer`。
- 符合 `12`、`- 12 -`、`第12页`、罗马数字等格式，候选 `page_number`。
- 多页中位置相近且文本相似，强烈判定为 `header` 或 `footer`。

注意：

- 杂志页顶部的大标题不能因为靠上就误判为页眉。
- 教材页底部的图注不能因为靠下就误判为页脚。
- 因此页眉页脚修正必须结合块大小、文本长度、重复性和模型类型。

### 7.6 标题修正

标题候选规则：

- 模型类型为 `title`。
- 文本块高度明显大于正文平均高度。
- 文本较短。
- 位于页面上半部分。
- 横向居中或跨越较宽区域。
- 符合章节编号格式，例如 `第一章`、`1.1`、`Chapter 2`。

降级规则：

- 如果候选标题位于页眉区域且多页重复，则改为 `header`。
- 如果候选标题靠近图片且以“图”“表”“Figure”“Table”开头，则改为 `caption`。
- 如果候选标题是广告、栏目名、装饰文字，杂志复杂页中可标为 `adornment` 或 `sidebar`。

### 7.7 图注和表注修正

图注/表注通常有以下特征：

- 靠近 `figure` 或 `table` 区域。
- 文本较短。
- 常以 `图`、`表`、`Figure`、`Fig.`、`Table` 开头。
- 字号通常小于正文或接近正文。

处理建议：

- 将图注绑定到最近的图片或表格。
- 如果图片进入 EPUB，图注跟随图片。
- 如果图片被摘要替代，图注和摘要合并处理。

### 7.8 阅读顺序排序

阅读顺序不是简单按 `y1` 排序。

#### 单栏页面

排序规则：

```text
从上到下，y1 小的先读
如果 y 坐标接近，再按 x1 从左到右
```

#### 双栏论文页面

排序规则：

```text
标题/摘要跨栏内容先读
左栏正文从上到下
右栏正文从上到下
图表和图注按其锚定位置插入
页眉页脚页码不进入正文顺序
```

#### 教材页面

排序规则：

```text
主标题
主正文
例题/提示框/边栏
图表与图注
页脚注释
```

边栏不一定直接插入正文，可以标记为 `reading_group: "sidebar"`，后续导出时决定是否放到正文或附注。

#### 杂志页面

杂志页排序最复杂，规则建议：

```text
主标题
副标题/导语
主正文栏
图片说明
侧栏/补充说明
装饰文字默认不进入正文
```

如果满足复杂页条件，不强行用规则决定最终顺序，而是进入第三层大模型兜底。

### 7.9 reading_group 设计

建议给每个块增加 `reading_group`：

| reading_group | 含义 |
|---|---|
| `main` | 主阅读流，默认进入正文 |
| `caption` | 图注/表注 |
| `sidebar` | 边栏补充内容 |
| `note` | 脚注、注释 |
| `noise` | 页眉页脚页码等噪音 |
| `visual` | 图片、表格、地图 |
| `unknown` | 不确定 |

这样后续 EPUB 导出可以选择：

- 只导出 `main`。
- 把 `caption` 插入图片附近。
- 把 `sidebar` 放到附注或单独小节。
- 默认过滤 `noise`。

### 7.10 复杂度评分

复杂度评分用于决定是否调用第三层视觉大模型。

建议指标：

| 指标 | 触发含义 |
|---|---|
| 图片面积占比高 | 杂志、地图、照片页常见 |
| 文本块数量过多 | 可能是复杂图文混排 |
| 检测出三栏以上 | 规则排序不稳定 |
| 大量 `unknown` | 模型不确定 |
| 大量块重叠 | 版面复杂或检测错误 |
| 标题/正文/图注关系混乱 | 需要语义判断 |
| 规则排序置信度低 | 需要大模型辅助 |

建议第一版规则：

```text
complexity_score =
  0.25 * image_area_score
  + 0.20 * column_score
  + 0.20 * unknown_score
  + 0.15 * block_count_score
  + 0.10 * overlap_score
  + 0.10 * low_confidence_score
```

分级：

| level | score | 处理方式 |
|---|---|---|
| `low` | 0.00-0.35 | 不调用大模型 |
| `medium` | 0.35-0.65 | 规则处理，保留低置信度标记 |
| `high` | 0.65-1.00 | 调用视觉大模型 |

### 7.11 第二层输出要求

第二层输出必须满足：

- 每个块有统一 `role`。
- 每个块有 `bbox`。
- 每个块有 `order`。
- 每个块有 `reading_group`。
- 页面有 `layout_type`。
- 页面有 `complexity`。
- 是否调用大模型由 `complexity.need_vlm` 明确标记。

## 8. 第三层：视觉大模型兜底

### 8.1 调用条件

只有满足以下条件之一才调用：

- `complexity.level = high`
- `layout_type = mixed_complex`
- `layout_type = multi_column` 且规则排序置信度低
- 图片面积占比超过 40%，同时存在多个文本块
- 大量 `unknown` 块
- 用户指定对某一页进行精修

### 8.2 输入给大模型的内容

不要只把页面图片丢给大模型。应该同时提供：

- 页面图片。
- 第一层检测框结果。
- 第二层规则初判结果。
- 希望大模型输出的固定 JSON 格式。

输入内容示意：

```json
{
  "page_no": 1,
  "image": "page_001.png",
  "detected_blocks": [
    {
      "block_id": "p001_b0001",
      "role": "title",
      "bbox": {"x1": 120, "y1": 80, "x2": 1800, "y2": 180}
    }
  ],
  "task": "判断这些版面块的阅读顺序，并指出哪些是正文、图注、侧栏或装饰文字。"
}
```

### 8.3 大模型输出

要求大模型只输出结构化 JSON，不输出自由散文。

建议输出：

```json
{
  "page_no": 1,
  "reading_order": [
    "p001_b0001",
    "p001_b0004",
    "p001_b0005"
  ],
  "role_corrections": [
    {
      "block_id": "p001_b0006",
      "from": "body",
      "to": "caption",
      "reason": "靠近图片且内容为图片说明"
    }
  ],
  "exclude_blocks": [
    {
      "block_id": "p001_b0009",
      "reason": "装饰性栏目文字，不属于正文"
    }
  ],
  "visual_summaries": [
    {
      "block_id": "p001_b0003",
      "summary": "本图展示了某地区山地与河谷地貌的空间关系。"
    }
  ],
  "confidence": 0.78
}
```

### 8.4 合并大模型结果

第三层输出不能无条件覆盖第二层。建议合并策略：

- 只允许修正 `role`、`order`、`reading_group`、`notes`、`visual_summary`。
- 不允许大模型修改原始坐标。
- 如果大模型返回的 `block_id` 不存在，丢弃该项并记录 warning。
- 如果大模型置信度低于阈值，只记录建议，不覆盖规则结果。
- 所有大模型修正都在块中标记 `source: "vlm_corrected"`。

合并后输出仍写回：

```text
layout/page_001.json
```

下游不需要关心这个页面是否调用过大模型。

## 9. 下游模块如何使用 layout JSON

### 9.1 OCR 模块

OCR 模块应该优先对以下区域识别文字：

```text
title
subtitle
body
caption
sidebar
note
```

默认不 OCR 或不进入正文：

```text
figure
table
header
footer
page_number
adornment
```

但 `figure`、`table` 可以在 P2 阶段交给视觉模型摘要。

### 9.2 文本清洗模块

清洗模块根据 `role` 和 `reading_group` 做处理：

- `noise` 默认过滤。
- `main` 合并为正文段落。
- `caption` 插入到相关图片或表格附近。
- `sidebar` 可作为附注或补充阅读。
- `unknown` 保留但标记低置信度。

### 9.3 EPUB 导出模块

EPUB 导出时建议：

- `title` / `subtitle` 用于章节识别。
- `body` 进入正文。
- `caption` 进入图注位置。
- `sidebar` 可以作为“补充阅读”段落。
- `figure` / `table` 如果有摘要，则插入摘要；如果没有摘要，则插入占位说明。

## 10. 开发实施顺序

### 第 1 步：定义统一 layout JSON

先固定数据结构，不急着接复杂模型。

完成标准：

- 有页面级字段。
- 有区块级字段。
- role、reading_group、layout_type、complexity 枚举固定。

### 第 2 步：接入 PP-DocLayout 原始输出

完成内容：

- 读取 `pages/page_001.png`。
- 运行版面检测。
- 保存 `layout_raw/page_001.json`。
- 生成调试图 `debug/page_001_layout_overlay.png`。

完成标准：

- 能看到每页检测框。
- 原始结果可复现。

### 第 3 步：做类型映射和坐标标准化

完成内容：

- 把模型输出映射成内部 `role`。
- 坐标统一成 `x1/y1/x2/y2`。
- 每个块生成稳定 `block_id`。

完成标准：

- 输出 `layout/page_001.json`。
- 每个块类型和坐标清楚。

### 第 4 步：实现单栏/双栏/多栏判断

完成内容：

- 根据正文块中心点判断栏位。
- 给每个正文块分配 `column`。
- 页面输出 `layout_type`。

完成标准：

- 单栏教材页判断正确。
- 双栏论文页判断正确。
- 杂志复杂页能标记为 `multi_column` 或 `mixed_complex`。

### 第 5 步：实现页眉页脚页码修正

完成内容：

- 根据坐标、文本特征、多页重复性修正 `header/footer/page_number`。
- 标记 `reading_group: "noise"`。

完成标准：

- 常见页码不会进入正文。
- 多页重复页眉能被识别。

### 第 6 步：实现阅读顺序排序

完成内容：

- 单栏排序。
- 双栏排序。
- 跨栏标题优先排序。
- 图表和图注绑定。

完成标准：

- 教材页阅读顺序基本正确。
- 论文页能先左栏后右栏。
- 杂志复杂页不强行排序，低置信度时进入复杂页流程。

### 第 7 步：实现复杂度评分

完成内容：

- 计算图片面积占比。
- 计算栏数。
- 统计 unknown 和低置信度块。
- 输出 `complexity.need_vlm`。

完成标准：

- 普通教材页不调用大模型。
- 普通论文页不调用大模型。
- 中国国家地理复杂页面能被标记为需要大模型。

### 第 8 步：接入视觉大模型兜底

完成内容：

- 只对 `need_vlm = true` 的页面调用。
- 请求中携带图片和检测框。
- 大模型返回阅读顺序、角色修正、图片摘要。
- 合并到最终 layout JSON。

完成标准：

- 复杂杂志页的主阅读顺序明显改善。
- 大模型修正可追踪。
- 失败时不影响普通流程。

## 11. 测试样例设计

建议准备以下样例：

| 样例 | 检查重点 |
|---|---|
| 单栏中文教材页 | 标题、正文、页码、图注 |
| 双栏英文论文页 | 双栏顺序、图表、参考文献 |
| 双栏中文论文页 | 中文 OCR 区块、摘要、脚注 |
| 中国国家地理普通文章页 | 多栏正文、图片、图注 |
| 中国国家地理复杂跨图页 | 大图、侧栏、装饰文字、复杂顺序 |

每个样例至少检查：

- 检测框是否覆盖主要内容。
- `role` 是否基本正确。
- `layout_type` 是否合理。
- `order` 是否符合阅读顺序。
- 页眉、页脚、页码是否标为噪音。
- 复杂页是否正确触发大模型。

## 12. 第一版验收标准

第一版版面分析模块完成后，应达到：

- 能读取页面图片并生成 `layout_raw`。
- 能生成统一 `layout` JSON。
- 能识别标题、正文、图片、表格、图注、页眉、页脚、页码。
- 能判断单栏、双栏、多栏或复杂混排。
- 能给正文块分配阅读顺序。
- 能判断复杂页是否需要大模型。
- 能输出调试图片，方便人工看检测框。

不要求第一版完成：

- 完美识别所有杂志装饰文字。
- 完美恢复所有跨页图文关系。
- 完美还原表格和公式。
- 完全不需要人工检查。

## 13. 推荐最终架构

建议把版面分析模块设计成一个独立 pipeline：

```text
LayoutPipeline
  |
  +-- DetectorLayer
  |     负责 PP-DocLayout 原始检测
  |
  +-- RuleLayer
  |     负责类型修正、栏位判断、阅读顺序、复杂度评分
  |
  +-- VlmLayer
  |     负责复杂页顺序修正和视觉摘要
  |
  +-- ExportLayer
        负责写出 layout JSON 和调试图片
```

下游 OCR、文本清洗、EPUB 导出都只读取 `layout/page_xxx.json`。

这样做的好处是：

- 第一层模型可以替换，不影响下游。
- 第二层规则可以持续迭代。
- 第三层大模型可以按需开启，成本可控。
- 每一步都有中间产物，方便调试复杂页面。

## 14. 当前阶段建议

当前最适合先实现到第二层：

```text
PP-DocLayout 检测
  -> 类型映射
  -> 坐标标准化
  -> 单栏/双栏/多栏判断
  -> 页眉页脚页码修正
  -> 阅读顺序排序
  -> 输出 layout JSON 和调试图
```

视觉大模型先预留接口和数据字段，不急着接入。等你用《中国国家地理》样例跑出一批失败页面后，再根据失败类型决定大模型提示词和合并策略。
