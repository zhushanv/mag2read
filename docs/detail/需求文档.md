智能杂志排版清洗与AI导读系统 需求规格说明书（初稿）

项目名称：Mag2Read —— 智能杂志文章电子化与导读系统
文档版本：V1.0
撰写日期：2026-06-25

---

1. 项目概述

1.1 项目背景

学生群体和研究人员经常需要阅读杂志、期刊上的长文，但纸质版不便携带，扫描版PDF在电子阅读器上又存在排版混乱（多栏交错）、页眉页脚干扰、图表看不清等痛点。同时，在课设层面需要一个能体现AI落地能力的完整项目。

1.2 项目目标

· 开发一个Web系统，用户上传杂志扫描件（PDF/图片）后，能自动提取正文，去除噪音，输出适合电子阅读器阅读的EPUB/Markdown文件。
· 利用AI大模型生成文章摘要和关键词，辅助快速阅读。
· 作为课程设计，展示“图像处理+AI模型+前后端开发”的综合能力。

1.3 用户角色

· 普通用户：上传文件、查看处理进度、下载结果。
· 管理员（课设加分项）：查看系统日志、管理API密钥。

2. 功能性需求

2.1 核心处理流程（后端）

模块 功能描述 对应技术/工具 优先级.  
文件上传与管理 支持批量上传PDF、JPG、PNG（单文件≤50MB），显示上传进度与文件列表。 前端表单 + Nginx静态服务 P0.  
智能版面分析 自动识别扫描件中的双栏/单栏布局，区分标题、正文、图注、页眉、页脚、页码。 PaddleOCR（开启layout_detection） P0.  
文字识别（OCR） 提取各区块内的手写/印刷体文字，输出带坐标信息的识别结果（JSON格式）。 PaddleOCR P0.   
排版噪音清洗 去噪：根据坐标规则过滤页眉、页脚、页码；拼段：跨页文字自动拼接，还原完整段落。 自定义Python规则引擎（正则+坐标逻辑） P1.   
图文分离与摘要 识别正文中的图片/表格位置，调用大模型生成文字描述摘要插入文中，代替原图。 Qwen-VL 或 GPT-4o-mini API P2.   
格式导出 导出为 EPUB（标准电子书格式，含章节元数据）和 Markdown（便于编辑）。 EbookLib / pandoc P0.  
AI导读生成 基于清洗后的全文，调用大模型生成：200字核心摘要 + 3~5个关键词，写入EPUB元数据。 DeepSeek API（便宜，上下文长） P1.   

2.2 交互与可视化（前端）

功能点 描述 优先级
拖拽上传 支持拖拽或点击上传，简洁美观的UI。 P0
实时进度流水线 展示“版面分析中 → 识别中 → 清洗中 → 生成导读中”各阶段进度（WebSocket推送）。 P1
双栏对比预览 页面左右分栏，左显示杂志原图（高亮识别区域），右显示清洗后的纯文本。点击左边区域，右边文字自动定位。 P2（课设亮点）
下载与历史记录 提供下载按钮；本地存储处理记录（至少保留最近5条）。 P1

3. 非功能性需求

· 性能要求：单篇20页杂志处理总时长 ≤ 3分钟（不含大模型API调用等待时间）。
· 稳定性：支持并发处理不超过3个任务（防内存溢出）。
· 可扩展性：OCR引擎和AI模型支持插拔式更换（配置文件切换）。
· 易用性：提供简单的操作指引，无复杂配置。

4. 技术栈建议

本项目采用前后端分离架构。

· 前端：React + TypeScript + Vite。  
· UI：Tailwind CSS + shadcn/ui + lucide-react。  
· 页面预览与版面框渲染：react-konva，用于在页面图片上叠加 layout block、OCR 坐标框和点击交互。  
· 前端状态管理：TanStack Query 负责接口请求和任务状态轮询，Zustand 负责页面选择、预览状态等局部交互状态。  
· 后端：Python + FastAPI，负责上传、任务查询、页面预览、结果下载等接口。  
· 异步任务：Redis + Celery，OCR、版面分析、导出等耗时流程放入 Worker 执行，避免阻塞 Web 请求。  
· 数据库：MySQL，保存任务、文件、页面、处理步骤和导出记录等元数据。  
· 文件存储：本地 `backend/storage/tasks/{task_id}/` 目录，保存原始文件、页面图片、OCR JSON、清洗结果和导出文件。后续可替换为 MinIO 或对象存储。  
· AI/OCR 核心：PaddleOCR、PP-DocLayout、PyMuPDF、Pillow；AI 导读后续可接入 DeepSeek API。  
· 部署：第一阶段本地运行，后续可使用 Docker Compose 统一管理前端、FastAPI、Worker、MySQL 和 Redis。

数据库设计原则：

· MySQL 不直接保存 PDF、图片、EPUB、OCR JSON 等大文件，只保存这些文件的路径和处理状态。  
· `tasks` 表保存任务主状态，例如 pending、processing、success、failed。  
· `task_files` 表保存输入文件、页面图片、OCR JSON、Markdown、EPUB 等文件索引。  
· `task_pages` 表保存每一页的宽高、质量状态、页面类型、OCR 置信度和是否需要人工检查。  
· `task_steps` 表保存 render、layout_detect、layout_refine、ocr、text_cleaning、document_build、export、ai_reading 等阶段的状态和耗时。  
· `export_records` 表保存导出文件记录，例如 EPUB、Markdown、DOCX、HTML、TXT。  
· 如果第一阶段不做登录，`users` 表可以暂缓实现，但 `tasks` 表可预留 `user_id` 字段，方便后续扩展。

5. 接口设计（API 预览）

方法 端点 描述
POST /api/upload 上传文件，返回 task_id
GET /api/task/{task_id}/status 获取处理状态与进度
GET /api/task/{task_id}/result 下载转换后的文件
WS /ws/task/{task_id} WebSocket 实时推送进度
