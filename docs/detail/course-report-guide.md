## 2. 摘要写法

摘要应避免空泛描述，可以突出实际问题和解决方法：

```text
针对扫描版 PDF 难以复制、搜索和二次编辑的问题，本文设计并实现了一个基于 Python 与 React 的智能 PDF OCR 转换系统。系统以后端 FastAPI 服务为核心，结合 PyMuPDF、OpenCV 和 PaddleOCR 完成 PDF 页面渲染、图像预处理和文字识别，并通过版面分析和阅读顺序恢复将 OCR 结果重建为结构化文档。前端采用 React 实现文件上传、任务进度展示、转换历史和结果下载。系统支持将扫描版 PDF 导出为 DOCX、EPUB、HTML、Markdown 和 TXT 等格式。实验结果表明，该系统能够较好处理常见教材、论文和扫描资料，在中文识别、段落恢复和多格式导出方面具有较好的实用价值。
```

## 4. 答辩演示流程

建议控制在 5 到 8 分钟。

### 第一步：说明问题

展示一个扫描版 PDF，说明它不能复制文字，也不方便搜索和编辑。

### 第二步：展示系统首页

说明系统支持上传 PDF、选择导出格式、启用 OCR 和版面分析。

### 第三步：执行转换

上传一份扫描 PDF，选择 DOCX 或 EPUB，开始转换。

### 第四步：展示任务进度

展示异步任务状态、进度条和 OCR 质量指标。

### 第五步：展示转换结果

打开导出的 Word 或 HTML 文件，展示识别出的文字、段落、标题、图片或表格。

### 第六步：说明技术亮点

重点讲：

- 扫描 PDF 页面渲染。
- 图像预处理。
- PaddleOCR 中文识别。
- 版面分析。
- 中间文档模型。
- 多格式导出。
- React + FastAPI 全栈架构。

## 5. 高分亮点包装

### 5.1 不只是调用 OCR

应强调系统做了完整流程：

```text
PDF 渲染 → 图像增强 → OCR 识别 → 版面分析 → 文档重建 → 多格式导出
```

### 5.2 引入中间文档模型

这是软件工程亮点。可以说明：

- OCR 模块只负责识别。
- 文档模型负责统一结构。
- 导出模块只依赖文档模型。
- 后续增加新格式更容易。

### 5.3 有质量评估

可以展示：

- 平均置信度
- 低置信度文本数
- 页面处理耗时
- 文本块数量
- 表格数量

### 5.4 有真实应用价值

强调日常场景：

- 扫描教材转 Word。
- 课程资料转可搜索文本。
- 论文扫描件转 Markdown 笔记。
- PDF 资料转 EPUB 阅读。

## 7. 项目不足写法

报告里不要回避限制，可以这样写：

```text
由于 PDF 文件本身缺少明确的语义结构，扫描版 PDF 又只能通过图像识别恢复内容，因此系统暂时无法保证对所有复杂艺术排版进行完全还原。对于低分辨率、严重倾斜、背景复杂或手写体文档，OCR 准确率仍会受到影响。后续可以引入更强的版面检测模型和人工校对模块，提高复杂场景下的转换质量。
```

这种写法比简单说“效果不好”更专业。

## 8. 答辩可能问题

### 为什么不用现成 PDF 转换软件？

本项目不是单纯复刻商业软件，而是围绕扫描版 PDF 的 OCR 识别流程进行工程实现，重点体现页面渲染、图像预处理、文字识别、版面分析和多格式导出之间的完整链路。

### 为什么需要异步任务？

OCR 转换耗时较长，如果直接在接口中处理，会导致请求阻塞甚至超时。使用异步任务后，前端可以实时查询进度，后端也能更稳定地处理大文件。

### 为什么要设计中间文档模型？

不同导出格式需要的数据结构不同。如果 OCR 结果直接导出，会导致代码耦合严重。中间文档模型可以统一表达标题、段落、图片、表格等结构，使 DOCX、EPUB、HTML 等导出模块可以复用同一套数据。

### 复杂杂志排版怎么处理？

系统先通过 OCR 获取文字框坐标，再根据文本框位置进行区域划分和阅读顺序恢复。对于多栏页面，系统会按列聚类后在每列内部排序。对于特别复杂的艺术排版，系统会尽量恢复可阅读结构，但不保证像素级还原。
